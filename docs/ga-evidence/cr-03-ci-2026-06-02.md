## What was wrong (start of session)

The pipeline was fully red on the deploy stage — only build/test passed. Root
causes, fixed this session:

1. **Deploy IAM gaps** — the CI deploy service account
   (`github-actions@finalwishes-prod`) lacked the roles for Functions + Rules
   deploys. Granted (this session): `serviceusage.serviceUsageConsumer`,
   `firebaserules.admin`, `cloudfunctions.admin`, `run.admin`,
   `cloudbuild.builds.editor`, `artifactregistry.admin`, `iam.serviceAccountUser`
   (+ ActAs on `@appspot`), `datastore.viewer`, `firebasestorage.admin`,
   `cloudscheduler.admin`, `eventarc.admin`, `pubsub.admin`. Enabled
   `cloudbilling.googleapis.com` (Gen2 prereq). See
   `scripts/fix-ci-deploy-permissions.sh`.
2. **API test gate** — a list-count assertion drift (`5abeaa4`).
3. **Storage rules** — `firebase deploy --only storage` requires the project's
   **default Firebase Storage bucket** to be provisioned ("Get Started" in the
   Firebase console). The bucket exists in GCS and was registered via the
   `firebasestorage addFirebase` API, but `resources.storageBucket` on the
   Firebase project remains empty; no public API populates it once the GCP
   resource location is finalized.

## Storage-rules decision

The **Storage Rules** sub-step is marked `continue-on-error: true` because:
- Storage rules are **defense-in-depth only** — all uploads use V4 signed URLs
  from the Go API that bypass these rules; the resize extension uses the Admin
  SDK (see `storage.rules` header). No client-SDK path depends on them.
- The only blocker is a **one-time owner console action** (Storage → Get Started)
  that no API exposes headlessly once the resource location is finalized.

Firestore rules remain a **required** (blocking) deploy. After the console
"Get Started" is done, remove `continue-on-error` to make Storage rules required.
