# Document Vault — Developer README

## Architecture

The Document Vault uses a **hybrid Cloud Storage + Firestore** architecture:

1. **Files** are stored in Google Cloud Storage (`finalwishes-vault` bucket)
2. **Metadata** is stored in Firestore at `estates/{estateId}/documents`
3. **Signed URLs** are generated by the Go API for both upload and download

## Upload Flow

```
User drops file
  → Client calls Go API: generateUploadUrl(estateId, fileName, contentType)
    → Go API generates signed PUT URL (15 min) from Cloud Storage
  → Client PUTs file directly to Cloud Storage via XHR (with progress tracking)
  → Client calls createDocumentRecord() to write metadata to Firestore
    → Firestore onSnapshot auto-refreshes the document list in UI
```

## Download Flow

```
User clicks "Download"
  → Client calls GET /api/v1/documents/download-url?storageKey=...
    → Go API generates signed GET URL (1 hour) from Cloud Storage
  → Client opens URL in new tab
```

## Firestore Schema

Collection: `estates/{estateId}/documents/{docId}`

| Field | Type | Description |
|-------|------|-------------|
| `originalName` | string | Original filename |
| `displayName` | string | User-facing name |
| `mimeType` | string | MIME type |
| `fileSize` | number | Size in bytes |
| `storageKey` | string | Cloud Storage object path |
| `storageBucket` | string | Bucket name |
| `folderId` | string | Category (Legal/Financial/Personal/General) |
| `tags` | string[] | Searchable tags |
| `version` | number | Always 1 (versioning deferred) |
| `status` | string | `active` or `archived` |
| `uploadedBy` | string | Firebase Auth UID |
| `createdAt` | timestamp | Server timestamp |
| `updatedAt` | timestamp | Server timestamp |

## File Validation

- Max size: 50 MB
- Allowed types: PDF, JPEG, PNG, HEIC, WebP, DOC, DOCX, TXT
- Category auto-inferred from filename keywords

## Security

- Upload URLs expire in 15 minutes
- Download URLs expire in 1 hour
- Both endpoints require Firebase Auth (JWT in Authorization header)
- Cloud Storage access is exclusively via signed URLs — no direct bucket access from client
- Firestore security rules enforce estate-scoped access
